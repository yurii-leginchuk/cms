## Defects ‚Üí instructions for chief-architect (prioritized)

### 1. CRITICAL ‚Äî GSC dimension filters silently dropped (whole-site data returned for every filtered query)
- **Symptom:** Any `querySearchConsole` call with `filters` (branded/non-branded, per-page, per-country drilldowns) returns unfiltered, whole-site rows. `getFullPageAnalysis` page filters likewise return whole-site numbers labeled as page-specific.
- **Evidence:**
  - Test #10: branded regex call and non-branded regex call both returned `rowCount: 625` (identical). Assistant reported non-branded = 4 clicks/6 impr (false).
  - Test #11: `filters:[{dimension:'page',operator:'equals',expression:'.../local-seo/'}]` returned **50 rows / 276 clicks** instead of one page row.
  - Direct `gsc-direct query` with the regex/page filters also returned the full 625/57 rows ‚Äî confirming the payload shape, not the model, is wrong.
- **Root cause:** `backend/src/gsc/gsc.service.ts` `query()` (‚âà lines 214-219) posts `params` verbatim to the GSC `searchAnalytics/query` endpoint. The Search Console API ignores a top-level `filters` field; it requires `dimensionFilterGroups: [{ groupType: 'and', filters: [...] }]`. `GscQueryParams` (lines 16-29) defines `filters` but it is never transformed. (Secondary: `searchType` should be mapped to the API's `type` field; harmless today since 'web' is default.)
- **Exact fix instruction:** In `query()`, build the GSC request body explicitly rather than spreading `params`. Map `params.filters` ‚Üí `dimensionFilterGroups: [{ filters: params.filters }]` (omit the key entirely when empty), and pass `startDate, endDate, dimensions, rowLimit, startRow, dimensionFilterGroups, type, aggregationType`. Add a regression test that asserts a `page equals X` filter returns ‚â§ the unfiltered row count and that branded+non-branded click sums reconcile to the total. Note: this also invalidates existing `gsc_cache` rows keyed on the old (unfiltered) responses ‚Äî call `clearCache()` or bump the cache-key version after the fix.

### 2. MODERATE ‚Äî Optimize-page workflow skips `proposePageContent`
- **Symptom:** "Optimize this page fully" produced the entire rewrite (meta, content, JSON-LD, internal links, recommendations) as inline chat text, never calling `proposePageContent`. No structured/saveable proposal; no automatic meta-length / placeholder / schema validation runs.
- **Evidence:** Test #12 ‚Äî TOOLS = `getFullPageAnalysis, searchPagesByKeyword`; PROPOSALS = 0; full deliverable present only as prose.
- **Root cause:** Workflow STEP 6 in `backend/src/prompts/prompts.service.ts` (`agent_optimize_page`, lines 61-63) and the system-prompt rule in `backend/src/agent/agent.service.ts` (line 225) instruct "ALWAYS call proposePageContent" but the model is satisfied by writing inline. The instruction is not enforced strongly enough relative to the inline –ë–£–õ–û/–°–¢–ê–õ–û display rules.
- **Exact fix instruction:** Strengthen STEP 6 to: "You MUST end by calling `proposePageContent`. Do not output the full rewrite as chat text instead of the tool call ‚Äî the inline summary must be brief and the canonical deliverable is the tool payload." Consider gating: if a workflow intent = optimize and no proposal tool was called, the system prompt should treat the turn as incomplete. Re-test #12 expecting PROPOSALS ‚â• 1.

### 3. MODERATE ‚Äî Validation warnings ignored on final output (meta over length shipped)
- **Symptom:** Proposed homepage meta description = 158 chars; `validation.valid = false` with the ‚â§155 warning; assistant delivered it as the final answer without shortening.
- **Evidence:** Test #14 ‚Äî `proposeMetaUpdate` output `validation: { valid:false, warnings:['Proposed description is 158 chars ‚Ä¶'] }`, yet presented as the recommendation.
- **Root cause:** `proposeMetaUpdate` / `validateProposedContent` correctly emit warnings, but nothing forces the model to act on a non-valid result. System prompt mentions "Shorten it and call this tool again" only inside the content-format block.
- **Exact fix instruction:** Add an explicit IMPORTANT RULE in `agent.service.ts`: "If any proposal tool returns `validation.valid === false`, you MUST revise and call the tool again before presenting it; never present an invalid proposal as final." Keep the hard caps at ‚â§60/‚â§155 in `proposal-validation.ts` (already correct).

### 4. MINOR ‚Äî Wrong tool for "performance fixes with estimated savings"
- **Symptom:** Homepage perf request answered via `getFullPageAnalysis` (DB scores), then invented percentage savings (10-20% CTR, 5-10% traffic) instead of real PSI savings-ms from failing audits.
- **Evidence:** Test #23 ‚Äî TOOLS = `getFullPageAnalysis`; no `analyzePageSpeed`; savings were ungrounded percentages.
- **Root cause:** Tool-routing in `agent.service.ts` (PageSpeed block, lines 241-245) ‚Äî `analyzePageSpeed` is the right tool for "concrete fixes / estimated savings" for a single URL, but the model defaulted to the broader page-analysis tool.
- **Exact fix instruction:** Add a routing line: "For 'analyze the performance of <one URL>' or 'fixes with estimated savings', call `analyzePageSpeed` (live audit with savings ms) ‚Äî `getFullPageAnalysis` returns DB scores only and must NOT be used to estimate performance savings." Reinforce the existing IMPORTANT RULE against inventing numbers (line 273) so percentage estimates aren't fabricated.
