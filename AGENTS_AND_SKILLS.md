# Агенты и навыки — справочник

Система из 9 специализированных агентов и 5 навыков, организованных по принципу **Research → Plan → Implement → Deploy → Monitor**. Они работают совместно: каждый предыдущий передаёт результат следующему.

---

## Агенты

### `@chief-architect` — Главный оркестратор

**Что делает:** Разбивает сложную задачу на части и координирует остальных агентов. Анализирует требования, собирает нужную команду, следит за качеством и собирает финальный результат.

**Когда использовать:** Задача затрагивает 3+ области — например, нужно добавить фичу с новым API-эндпоинтом, фронтенд-страницей и деплоем.

**Как поможет в этом проекте:**
- Когда нужно реализовать крупную фичу CMS (новый тип контента, интеграция с WordPress, SEO-модуль) — он соберёт правильную команду и не даст потерять детали при передаче между задачами.
- Поддерживает параллельный запуск агентов для независимых задач (например, бэкенд и фронтенд одновременно).

---

### `@docs-researcher` — Исследователь документации

**Что делает:** Ищет актуальную официальную документацию по библиотекам и API — с точными сигнатурами методов, версиями и источниками. Выдаёт структурированный **ResearchPack** за ~2 минуты.

**Когда использовать:** Перед реализацией чего угодно, что использует внешнюю библиотеку или API — особенно если давно не работал с ней или не уверен в версии.

**Как поможет в этом проекте:**
- TypeORM, NestJS, React Query — правильные декораторы и методы без гадания по памяти.
- При интеграции с WordPress REST API — точные эндпоинты и параметры.
- При обновлении зависимостей — находит breaking changes и путь миграции.

---

### `@implementation-planner` — Планировщик реализации

**Что делает:** Принимает ResearchPack и создаёт детальный план: какие файлы менять, в каком порядке, как проверять каждый шаг, как откатиться если что-то пошло не так.

**Когда использовать:** После исследования, перед началом кодинга — особенно когда изменение затрагивает несколько файлов или модулей.

**Как поможет в этом проекте:**
- Чёткий план с указанием файлов (`backend/src/modules/...`, `frontend/src/pages/...`) и порядком действий.
- Rollback-план через git — если что-то сломается, знаешь как вернуться.
- Принцип "минимальных изменений" — не трогает то, что не нужно трогать.

---

### `@code-implementer` — Исполнитель кода

**Что делает:** Выполняет план по шагам с обязательным TDD (сначала тест, потом код). При ошибке пытается исправить сам — до 3 попыток с разными стратегиями.

**Когда использовать:** После того как есть ResearchPack и план реализации. Это финальный шаг кодинга.

**Как поможет в этом проекте:**
- Реализует фичи строго по плану, не уходя в сторону.
- Пишет тесты к каждому изменению (NestJS unit/e2e тесты, Vitest для React).
- Самокоррекция — если тест упал, анализирует причину и пробует снова.

---

### `@brahma-analyzer` — Анализатор консистентности

**Что делает:** Проверяет, что спецификация, план и код согласованы между собой. Ставит оценку 0–100, блокирует переход к реализации если ниже 80.

**Когда использовать:** Между этапами — после планирования, перед кодингом. Также при ревью когда кажется что что-то не сходится.

**Как поможет в этом проекте:**
- Ловит расхождения до того как они станут багами: "план говорит одно, спека другое".
- Проверяет что API в плане соответствует тому что нашёл `@docs-researcher`.
- Особенно полезен при работе с WordPress-синхронизацией где много взаимозависимых частей.

---

### `@brahma-investigator` — Детектив по багам

**Что делает:** Системно расследует баги — идёт к корневой причине, а не к симптому. Использует трёхуровневое мышление (think / think hard / think harder) в зависимости от сложности. Три попытки с разными гипотезами.

**Когда использовать:** Когда баг непонятен, воспроизводится нестабильно, или обычные методы не помогают.

**Как поможет в этом проекте:**
- Разобраться в проблемах Docker Compose — почему сервис не стартует, почему связи между контейнерами не работают.
- Отладка TypeORM миграций и неожиданного поведения запросов.
- Странные ошибки в NestJS Guards, Interceptors, DI — там где трейс неочевиден.

---

### `@brahma-deployer` — Специалист по деплою

**Что делает:** Управляет продакшн-деплоями с canary-стратегией (5% → 25% → 50% → 100%). Автоматически откатывает если метрики выходят за пороги. Создаёт runbook-документацию.

**Когда использовать:** При деплое значимых изменений в продакшн.

**Как поможет в этом проекте:**
- Безопасный деплой с возможностью быстрого отката — актуально для CMS где важна доступность.
- Управление Docker Compose в продакшне — zero-downtime обновления.
- Проверки перед деплоем: тесты, security scan, staging валидация.

---

### `@brahma-monitor` — Специалист по наблюдаемости

**Что делает:** Настраивает три столпа наблюдаемости: метрики (Prometheus/Grafana), логи (ELK/Loki), трейсы (Jaeger). Создаёт алерты и SLO/SLI дашборды.

**Когда использовать:** При настройке мониторинга или когда нужно понять что происходит с сервисом в продакшне.

**Как поможет в этом проекте:**
- Настройка логирования для NestJS API — видеть медленные запросы и ошибки.
- Мониторинг синхронизации с WordPress — сразу знать если что-то упало.
- Алерты на ошибки в PostgreSQL запросах до того как пользователи заметят.

---

### `@brahma-optimizer` — Специалист по производительности

**Что делает:** Профилирует, находит узкие места и оптимизирует. Принцип: сначала измерить, потом оптимизировать — никаких догадок. Покрывает код, SQL, кэширование, масштабирование.

**Когда использовать:** Когда что-то работает медленно — API тормозит, страница долго грузится, запросы к БД тяжёлые.

**Как поможет в этом проекте:**
- Оптимизация TypeORM запросов — N+1 проблемы, неправильные джойны, отсутствующие индексы.
- Кэширование тяжёлых CMS запросов (списки постов, SEO метаданные).
- Анализ бандла фронтенда — что замедляет первую загрузку.

## Консультационный совет (Feature Advisory Board)

Три агента-эксперта, которых `@chief-architect` созывает **на этапе discovery** — *до* исследования и планирования — для любой новой фичи, связанной с SEO, аналитикой (Search Console) или встроенным AI-ассистентом. Они **только консультируют**: читают предложенную фичу и текущий код, затем возвращают приоритизированные требования (P0/P1/P2). **Код они не пишут.** Архитектор запускает их параллельно и сводит три заключения в единый **Requirements Digest**, который потом ведёт `@docs-researcher` и `@implementation-planner`.

---

### `@seo-specialist-advisor` — Голос SEO-специалиста

**Что делает:** Смотрит на фичу глазами элитного SEO с 15-летним опытом работы внутри Ahrefs, Semrush, Screaming Frog, Search Console. Проверяет, ложится ли фича на реальный рабочий цикл SEO: *найти возможность → оценить влияние → внести правку → проверить результат.*

**На что давит:** удобство рабочего процесса, доверие к данным (источник, период, свежесть), действенность (каждый инсайт → конкретное действие), массовые операции (фильтры, bulk-edit, экспорт), безопасные правки живого контента (review-before-apply, diff, undo).

**Как поможет в этом проекте:** прежде чем команда начнёт строить SEO-фичу (мета-редактор, striking-distance, анализ каннибализации, schema-модуль) — подскажет, как сделать её такой, чтобы SEO-специалист реально ей пользовался, а не бросил.

---

### `@data-analyst-advisor` — Голос дата-аналитика

**Что делает:** Смотрит на фичу глазами элитного аналитика (GA4, BigQuery, Looker, GSC API). Не доверяет ни одной цифре без происхождения. Проверяет корректность данных на этапе дизайна — самой дорогой ошибки продукта.

**На что давит:** определения метрик и единый источник истины, корректность дат/таймзон/периодов, lineage и свежесть (кэш GSC ~24ч, лаг данных), воспроизводимость (одинаковый вопрос → одинаковый ответ), сегментация и drill-down, экспорт сырых данных, честная агрегация (нельзя усреднять CTR и position) и визуализация.

**Как поможет в этом проекте:** ловит «дашборд врёт» до того как он построен — следит, чтобы метрики GSC/PSI/CrUX считались правильно, имели единое определение и были проверяемы.

---

### `@ux-designer-advisor` — Голос UX

**Что делает:** Смотрит на фичу глазами элитного продуктового UX-дизайнера для плотных профессиональных инструментов. Цель — ясность и минимум трения, а не украшательство.

**На что давит:** информационная архитектура и иерархия, реальный flow и снижение трения, прогрессивное раскрытие плотных данных, обратная связь системы (loading/streaming/async-джобы), состояния empty/error/edge, доверие к AI-фичам (review-before-apply, diff, undo, показ источников), консистентность с дизайн-системой (**shadcn/ui** в `frontend/src/components/ui`), доступность.

**Как поможет в этом проекте:** прежде чем рисовать UI, подскажет где фича должна жить, какие состояния обязательно спроектировать, и какие существующие компоненты переиспользовать вместо изобретения новых.

## Навыки

Навыки — это встроенные инструкции, которые автоматически активируются когда Claude работает с подходящей задачей. Не нужно их вызывать явно.

---

### `research-methodology` — Методология исследования

**Что делает:** Задаёт стандарт для поиска документации. Автоматически активируется когда задача требует использования внешней библиотеки. Определяет версию из `package.json`, выбирает официальные источники, требует цитировать каждое API с URL.

**Результат:** ResearchPack — структурированный документ с API-сигнатурами, примерами кода и источниками.

**Польза для проекта:** Каждый раз когда Claude помогает с NestJS, TypeORM, React Query или любой другой библиотекой — он сначала проверяет актуальную документацию под вашу версию, а не полагается на обучающие данные.

---

### `planning-methodology` — Методология планирования

**Что делает:** Устанавливает стандарт для создания планов реализации. Приоритет: минимальные изменения, расширение вместо модификации, каждое изменение должно быть обратимым.

**Результат:** План с указанием конкретных файлов, порядком шагов, методами проверки и rollback-процедурой.

**Польза для проекта:** Когда Claude предлагает реализацию — она будет хирургически точной. Не будет трогать лишние файлы, не будет ломать то что работает. Особенно важно для работы с NestJS модулями где легко сломать DI.

---

### `quality-validation` — Валидация качества

**Что делает:** Определяет оценочные рубрики и пороги для каждого этапа. ResearchPack должен набрать 80+/100, Plan — 85+/100. Если не прошёл — блокирует переход к следующему этапу с конкретным списком проблем.

**Результат:** Отчёт с оценкой, разбивкой по категориям и конкретными дефектами (не "могло быть лучше", а "API `foo()` не найден в официальных доках").

**Польза для проекта:** Предотвращает ситуацию "мусор на входе — мусор на выходе". Если исследование было плохим, план на его основе не пройдёт дальше. Меньше багов доходит до кода.

---

### `pattern-recognition` — Распознавание паттернов

**Что делает:** После успешных реализаций анализирует код и извлекает переиспользуемые паттерны. Сохраняет их в `knowledge-core.md`. Перед новой реализацией предлагает подходящие паттерны из истории.

**Результат:** Постепенно накапливаемая база паттернов, специфичных для проекта.

**Польза для проекта:** Со временем Claude всё лучше понимает как именно в этом проекте принято делать NestJS сервисы, React компоненты, TypeORM репозитории. Новые фичи будут следовать уже установленным паттернам, а не изобретать каждый раз заново.

---

### `context-engineering` — Управление контекстом

**Что делает:** Оптимизирует что попадает в контекстное окно. Архивирует устаревшую информацию в `knowledge-core.md`, убирает шум, оставляет только то что нужно для текущей задачи. Активируется автоматически в длинных сессиях (50+ сообщений).

**Результат:** 84% меньше токенов, 39% лучше качество ответов (по данным Anthropic).

**Польза для проекта:** Длинные сессии работы с CMS не деградируют со временем. Claude не начинает "забывать" начало разговора или повторять ошибки из-за замусоренного контекста.

## Как агенты работают вместе

Типичный сценарий для этого проекта — добавить новый тип контента в CMS:

```
Ты → @chief-architect
       ├── @docs-researcher → ResearchPack (TypeORM, NestJS)
       │       ↓ [quality-validation: 80+ check]
       ├── @implementation-planner → Plan
       │       ↓ [quality-validation: 85+ check]
       ├── @brahma-analyzer → Consistency report
       │       ↓ [если 80+]
       └── @code-implementer → Working code + tests
```

При баге:
```
Ты → @brahma-investigator → Root cause + fix
```

При медленном API:
```
Ты → @brahma-optimizer → Profile → Fix → Benchmark
```

## Как запускать

Агентов можно вызывать напрямую — просто упомяни имя в запросе:

```
"Используй @docs-researcher чтобы найти документацию по TypeORM для версии из package.json"

"@chief-architect, мне нужно добавить модуль для управления медиафайлами с загрузкой на S3"

"@brahma-investigator, NestJS Guard не срабатывает на этом роуте — помоги разобраться"
```

Навыки активируются автоматически — когда Claude видит что задача требует исследования документации или создания плана, он применяет соответствующую методологию без явного вызова.
